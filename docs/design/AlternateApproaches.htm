Daikon .NET Front End

1-20-11 Meeting Notes

 

Purpose: At this meeting the project architecture was reconsidered. A list of possible Rewriting libraries and alternate profiler architectures was brainstormed. In the end the Assembly.Load method, listed in the “Misc. other approaches” bullet was chosen, along with the CCI Metadata Rewriting library. The Assembly.Load method was chosen because it met all projects goals, worked well with a rewriting library, and was simple to test (and later build).

 

Possible Profiler Approaches:

·         Modify and run copy of original binary

                Theoretically, the Reflection API could be used to examine an assembly, and the Refelction.Emit API could be used to create a new binary with our instrumentation code added. It looks like the process of building a new assembly is fairly involved. If however, copying classes, methods, and fields can be done very generically the whole assembly could be copied.

·         Add prologues and epilogues through Profiler API entirely in unmanaged profiler

                Prologues and epilogues can be added to methods by modifying the IL right before the method is JIT compiled using Profiler API hooks. Prologues and epilogues can be generated by creating the byte codes necessary for the output function calls and inserting them at appropriate points in the method IL. This is difficult because it requires manually obtaining the references for all fields and methods. Also, there is a lot of function metadata that is difficult to update that must be maintained. IL rewriting libraries take care of this difficult work.

·         Add prologues and epilogues through Profiler API using IL rewriting library

                This is the approach I have spent the most time on. It is similar to the one above; the difference is that instead of the instrumentation being done at the byte level, an IL rewriting library creates new IL for us. The library adds the instrumentation calls necessary and communicates these changes to the profiler API which substitutes the original IL for the instrumented IL. The Profiler API prohibits direct calls to managed code, so the IL rewriter could either instrument the entire binary on startup and save the results to an intermediate location, or could receive an indirect signal from the profiler API about which methods needs to be instrumented, and then reply with the instrumented IL for that method. A few methods I’ve considered for this communication could be a pair of files or a socket. I hit a potential wall in this method that the method tokens generated by the IL rewriting library may not match the token from the original program. It’s possible this could be worked around by setting the profiler to only selectively use bytes from the IL rewriting library, not using the new bytes corresponding to method tokens.

·         Use IL rewriting library to generate new assembly

                This is the approach is similar to the above approach except without the use of the profiler API. The assembly is instrumented all at the beginning. Ideally the assembly can be run without saving it to disk. However, there doesn’t appear to be any way to do this using only .NET. When looking at rewriting libraries (below) I discovered RAIL may have the potential to run a modified assembly from memory. If this cannot be done the modified assembly would be saved to disk. The “launcher” program would then execute the modified assembly and likely delete it when finished.

·         Misc. other approaches

                There is an Assembly.Load(byte[]) if the modified assembly can be constructed in memory, i.e. as output from an IL rewriting library, but I haven’t explored this. Perhaps proxies could be created to provide the instrumentation calls. Here’s one project that implements proxies: http://www.castleproject.org/dynamicproxy/index.html but its class proxies only work on virtual members.

 

Existing CIL Rewriting Libraries:

·         Mono Cecil (http://www.mono-project.com/Cecil)

                This is an open source CIL rewriting project. It has the basic features expected for an IL rewriting library: loads managed assemblies, allows browsing of contained types, code modification and saving the assembly to disk. There’s not great documentation for it, so I can’t verify everything it can do. From investigating the source I believe it would be able to make add all the necessary instrumentation code. The requirement to save assemblies to disk however may contradict the project goal of avoiding disk modification.

·         PostSharp (http://www.sharpcrafters.com/)

                This is a commercial tool that assists developers with aspect oriented programming. It’s about $200 for a personal license, but it looks like there’s a community edition we may be able to distribute. It has nice hooks for method Entry and Exit that provides access to arguments. I’m not completely certain other information about the object calling the method is accessible however. Also, it looks like source code modifications are necessary to enable it to be run, something we would like to avoid.

·         Runtime Assembly Instrumentation Language (http://rail.dei.uc.pt/index.htm)

                This is another IL rewriting library that comes from a research project. It seems to implement all the standard features, from their paper it seems RAIL was the first general purpose instrumentation library for the .NET platform. Intriguingly however it looks like they expose the ability to run modified assemblies without saving to disk. I think this library definitely deserves a look if we keep the same architecture. There has been no work on the project since 2005 however.

.NET Profiling API (http://msdn.microsoft.com/en-us/library/ms404386.aspx)

                This is what I have been using so far. The API provides a variety of hooks, including function enter and exit. It also allows manual modification of function IL when a method is entered or exited. Manual modification is difficult because there’s a lot of function metadata that has to be updated. Among these are method and stack size which can be easily enough, but code labels and exception handlers are more difficult. It requires implementation in C++ and prohibits calls into managed code.

·         System.Reflection.Emit (http://msdn.microsoft.com/en-us/library/system.reflection.emit.aspx)

                This .NET API allows the creation of new assemblies, modules, methods, etc.  It can run these dynamic assemblies without saving them to disk. The drawback here is the entire program to be profiled would have to be recreated in memory. This may be possible using the Reflection API to load the program to be profiled, then iterating over its structures (modules, methods, member, etc.) and recreating them in the new assembly in parallel.

·         CCI Metadata (http://ccimetadata.codeplex.com)

                This is an MSR project allowing viewing and modification of .NET assemblies, modules, etc. Their API definitely allows all the features necessary for our instrumentation. Also, they claim significantly better performance than the .NET reflection APIs. I’ve written the instrumentation code to use this library. Unfortunately, it doesn’t allow the modified assembly to be run; the assembly must be saved to disk.
